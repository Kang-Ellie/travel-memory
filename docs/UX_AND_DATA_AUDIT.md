## 3. 데이터 연결 상태 점검 (장소·리뷰·사진·평가·버킷·금액·일정)

### 잘 되어 있는 것 ✅

| 연결 | 평가 |
|---|---|
| **장소 ↔ 방문(리뷰·사진)** | 구조적으로 올바름. 장소(place)와 방문(timeline_event)을 분리해서 리뷰·별점·사진이 **방문마다** 남고, 장소 상세에서 여러 여행에 걸친 방문 기록을 모아 보는 "족보" 개념이 스키마에 정확히 구현됨. 이 앱에서 가장 잘 설계된 부분 |
| **금액 ↔ 장소** | 지출→이벤트→장소 체인으로 장소별 누적 지출이 통화별로 집계됨 (`/api/places/:id/detail`). 이벤트를 삭제해도 지출은 `SET NULL`로 살아남아 가계부 총액은 보존 |
| **버킷 ↔ 일정** | 일정에 버킷을 연결하면 자동으로 done 처리 + 어느 여행에서 이뤘는지 기록 (`routes.ts` 이벤트 수정 시 bucket done). 버킷→장소(`linked_place_id`), 버킷→여행 연결 모두 존재 |
| **보관함 → 일정 편입** | SNS 링크/메모/이미지를 일정으로 변환하는 파이프라인이 완결적 (장소 생성 + 이벤트 생성 + 사진 이관 + 원본 삭제) |
| **삭제 안전장치** | 사용 중인 장소/동행인 삭제는 FK로 거부하고 친절한 에러 메시지. 여행 삭제 시 하위 데이터 CASCADE 일관성 있음 |
| **예산 ↔ 환율** | 여행별 고정 환율 등록 시 외화 지출도 원화 환산되어 예산 진행률에 합산됨 (`settlement.ts:98`). README의 "KRW만 집계" 문구가 오히려 구버전 — README 업데이트 필요 |

### 일정 관리 종합 평가

일차(day_number)+순번(sequence) 구조, 드래그 재정렬, 미배정 티켓(day_number NULL) → 일차 배정, 티켓과 방문지의 표시 순번 분리(공항·발렛·숙소는 번호에서 제외) 등 **핵심 흐름은 견고하게 잘 만들어져 있습니다.** 남은 위험은 위 5번(기간 축소)과 재정렬의 비원자성(성능 문서 A-2, 동시에 두 기기에서 드래그하면 순번 꼬임 가능) 두 가지가 실질적입니다.

> **한 줄 결론**: 장소-리뷰-사진-지출-버킷의 연결은 설계 자체가 좋고 실제로 잘 돌아갑니다. 문제는 연결이 아니라 **연결이 끊길 때(삭제·수정)의 뒷정리**입니다 — R2 고아, 버킷 done 롤백, 기간 축소, 지출의 족보 이탈 네 가지가 전부 "삭제/변경 시나리오"에 몰려 있으니, "지우기/줄이기 시나리오 일괄 정비"를 하나의 작업 단위로 처리하는 것을 추천합니다.
