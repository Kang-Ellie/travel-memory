# 트래블 온 — 프로젝트 객관 평가

> 2026-07 기준, 코드베이스 전수조사(서버 5개 파일 + 프론트 60여 개 컴포넌트, 약 12,000 LOC)를 바탕으로 작성.
> 칭찬용 문서가 아니라 "지금 어디에 서 있는지"를 있는 그대로 적은 문서입니다.
> 성능 개선 항목의 상세 리스트는 [PERFORMANCE_IMPROVEMENTS.md](./PERFORMANCE_IMPROVEMENTS.md) 참고.

---

## 1. 지금 수준은 어느 정도인가

### 규모와 완성도 (팩트)

| 항목 | 수치 |
|---|---|
| 총 코드 | 약 12,000 LOC (TS/TSX) |
| 프론트 컴포넌트 | 60여 개 |
| API 엔드포인트 | 약 80개 |
| DB 테이블 | 25개 + 상세 테이블(flight/valet/lodging/reservation) 1:1 설계 |
| 배포 | Cloudflare Pages + Railway + Neon + R2, 실제 운영 중 |
| 테스트 | **0개** |
| DB 인덱스 | **0개** (PK/UNIQUE 제외) |
| 트랜잭션 사용 | **0회** |

### 잘하고 있는 것 (구체적 근거와 함께)

- **끝까지 배포해서 실사용까지 갔다.** 대부분의 사이드 프로젝트는 로컬에서 죽습니다. 이 프로젝트는 R2 파일 스토리지 전환, CORS/쿠키 크로스 도메인 설정, PWA share_target까지 실전 배포 문제를 전부 통과했습니다. 이것 자체가 상위권 역량입니다.
- **도메인 모델링이 깊다.** "여행 앱"을 만들면 보통 trips/events/expenses 3개 테이블에서 끝나는데, 여기는 항공·발렛·숙소·예약을 이벤트와 1:1 상세 테이블로 분리했고(`db.ts:146`, `293-327`), 여행별 고정 환율, 미배정 티켓(day_number NULL 허용), 장소 족보(여러 여행에 걸친 누적 방문) 같은 실사용에서만 나오는 요구를 스키마에 정확히 반영했습니다.
- **주석의 질이 높다.** "무엇"이 아니라 "왜"를 적습니다. 예: multer의 latin1 한글 파일명 깨짐(`upload.ts:19`), Node 22의 unhandled rejection 프로세스 종료 방어(`index.ts:9`), 구글 API 요금 폭탄 방어용 일일 상한(`routes.ts:10`). 실제로 문제를 겪고 원인을 이해한 사람만 쓸 수 있는 주석입니다.
- **방어적 사고가 있다.** EXIF 파싱 실패 시 fail-open, 이미지 압축 실패 시 원본 유지, async 핸들러 에러 자동 next(err) 래핑(`routes.ts:46`), HMAC 서명 세션 토큰의 timingSafeEqual 사용(`auth.ts:23`). 주니어에게 기대하기 어려운 부분입니다.
- **과거 버그를 데이터 마이그레이션으로 복구한 흔적**(`db.ts:279-286`, `404-407`)이 있습니다. 실운영을 해봤다는 증거입니다.

### 부족한 것 (성장의 다음 단계이기도 함)

1. **테스트가 0개.** 특히 `settlement.ts`(정산 알고리즘)와 `dayISODate` 같은 날짜 계산은 돈과 직결되는 순수 함수인데 테스트가 없습니다. 지금 규모에서 리팩토링하면 뭐가 깨졌는지 알 방법이 수동 클릭뿐입니다.
2. **데이터 계층 규율 부재.** 인덱스 없음, 트랜잭션 없음, N+1 쿼리(이벤트 1개당 6쿼리, `routes.ts:743-765`), `initSchema`에 마이그레이션이 400줄 누적되어 서버 기동마다 전부 재실행.
3. **거대 파일.** `routes.ts` 1,554줄, `TripWorkspace.tsx` 1,189줄(EventCard 하나에 useState 30개+), `styles.css` 2,675줄. 기능이 추가될수록 수정 비용이 비선형으로 늘어나는 구조입니다.
4. **입력 검증 계층 없음.** `req.body as { ... }` 타입 단언만 있고 런타임 검증(zod 등)이 없어, 잘못된 요청이 그대로 DB까지 갑니다. `any`도 매퍼 전반에 퍼져 있습니다.
5. **React 최적화 관례 부재.** `useMemo`/`useCallback`/`React.memo`/`React.lazy` 사용이 코드베이스 전체에서 0회. 데이터 fetching도 화면마다 수동 `useEffect` + 전체 refetch 패턴입니다.
6. **CI/린트 없음.** 빌드가 `tsc --noEmit`뿐이고, push 시 자동 검증이 없습니다.

### 종합: 어느 레벨인가

**"제품을 만들어 배포하는 능력"은 미들급, "소프트웨어 엔지니어링 관례"는 주니어급.** 이 갭 자체는 자연스럽습니다 — 혼자 만드는 1인 프로젝트에서는 테스트·마이그레이션·코드 구조의 비용이 당장 안 보이기 때문입니다. 다만 이 프로젝트는 이미 그 비용이 보이기 시작하는 규모(12K LOC)에 도달했습니다. 지금이 관례를 도입하기 가장 좋은 시점입니다.

## 2. 이 프로젝트는 실제 사용성이 있는가

### 지금 형태(가족용 사설 앱)로는 — **있다, 이미 증명됨**

- 커밋 히스토리와 데이터 복구 마이그레이션이 실사용을 증명합니다.
- "비밀번호 1개 + 가족 공용"이라는 인증 모델은 이 용도에서는 **결함이 아니라 올바른 설계 판단**입니다. 요구사항에 맞는 최소 복잡도를 선택한 것.
- 경쟁 제품(트리플, 트라비포켓 등) 대비 차별점도 실재합니다: 장소 족보(여러 여행에 걸친 누적 기록), 발렛/항공/숙소 티켓 카드, SNS 공유 타겟으로 링크 수집 → 일정 편입 파이프라인. 이건 범용 앱이 못 하는 "우리 가족의 여행 방식"에 맞춘 기능입니다.

### 공개 서비스로 확장하려면 — **근본 재설계가 필요**

- 멀티테넌시가 없습니다. 모든 테이블에 소유자 개념이 없어서 사용자 계정을 붙이려면 스키마 전체에 손을 대야 합니다.
- 로그인 rate limit 없음, 비밀번호 비교가 타이밍 공격에 노출(`routes.ts:239`), 파일 경로 검증 없음 등 공개 전제가 아닌 보안 수준입니다.
- **권고: 공개 서비스화를 목표로 삼지 마세요.** 이 프로젝트의 가치는 "실사용 니치 제품을 끝까지 만든 경험"이고, 그건 포트폴리오로서 이미 완성형입니다. 억지로 SaaS로 늘리는 것보다, 아래 3장의 엔지니어링 품질을 끌어올리는 편이 배우는 것도 보여줄 수 있는 것도 많습니다.

## 3. 이 프로젝트에서 더 나아가야 할 것 (우선순위 순)

1. **정산·날짜 로직에 테스트 붙이기** — `settlement.ts`, `dayCount`/`dayISODate`/`todayDayNumber`는 순수 함수라 vitest로 30분이면 시작합니다. "테스트를 짜본 사람"과 아닌 사람의 갭이 채용 시장에서 가장 크게 보이는 지점입니다.
2. **성능 리스트 중 1순위 그룹 처리** — 인덱스 추가, N+1 제거, 20MB mp4 삭제, 이미지 lazy loading. ([상세](./PERFORMANCE_IMPROVEMENTS.md)) 총 하루 작업으로 체감 속도가 눈에 띄게 바뀝니다.
3. **마이그레이션 체계 도입** — `initSchema` 400줄을 node-pg-migrate 같은 도구로 옮기고, 기동 시 전체 DDL 재실행을 멈추기.
4. **routes.ts 분해** — 도메인별 라우터 파일(trips.ts, places.ts, expenses.ts…)로 나누고, `req.body as` 대신 zod 스키마 검증을 경계에 한 층 깔기.
5. **TripWorkspace 분해 + 데이터 계층 도입** — TanStack Query를 들이면 "수정 후 9개 API 전체 refetch" 패턴이 자연스럽게 사라집니다.
6. **CI 한 줄** — GitHub Actions에서 `tsc --noEmit` + vitest만 돌려도 됩니다.

## 4. 개발자로서의 평가와 성장 방향

### 코드에서 보이는 강점

- **제품 감각과 집요함.** 커밋 메시지("단청 마음에 안들어서 바꿈", "티켓 클릭 = 수정")에서 사용자 관점으로 계속 다듬는 태도가 보입니다. 이건 가르쳐서 되는 게 아닙니다.
- **문제의 뿌리를 찾는 습관.** 한글 파일명 깨짐을 latin1 디코딩으로 해결하고, 버그로 깨진 과거 데이터를 SQL로 복구하는 것은 증상이 아니라 원인을 다루는 방식입니다.
- **트레이드오프를 의식적으로 선택.** "비밀번호 1개 인증", "지도는 키 없어도 나머지 기능 동작" 같은 결정에 이유가 명시되어 있습니다.

### 보완하면 좋을 성향

- **"돌아가면 됨"에서 멈추는 경향.** 인덱스·트랜잭션·테스트 부재는 지식 부족이라기보다, 혼자 쓰는 규모에서는 안 아프기 때문에 넘어간 흔적으로 보입니다. 문제는 이 습관이 협업 환경에서는 그대로 약점이 된다는 것.
- **구조화 시점을 놓치는 경향.** 1,500줄 파일은 하루에 만들어지지 않습니다. "이 파일 좀 커졌네" 시점에 나누는 감각을 의식적으로 훈련할 필요가 있습니다.

### 성장 로드맵 제안

| 단계 | 무엇을 | 왜 |
|---|---|---|
| 지금 당장 | 이 프로젝트에 테스트·인덱스·마이그레이션 도입 | 새 프로젝트가 아니라 **살아있는 코드에 규율을 소급 적용하는 경험**이 실무와 가장 비슷함 |
| 1~3개월 | 관측성 경험: 서버에 구조화 로깅(pino), 에러 트래킹(Sentry 무료 티어) | "배포 후 무슨 일이 일어나는지 아는" 개발자가 되는 첫걸음 |
| 3~6개월 | 남의 코드 경험: 관심 있는 오픈소스에 작은 PR, 또는 스터디에서 코드 리뷰 주고받기 | 혼자 개발의 최대 사각지대는 "내 코드를 남이 읽는 경험"의 부재 |
| 면접 대비 | 이 프로젝트의 의사결정을 말로 정리 — "왜 R2로 옮겼나", "왜 비밀번호 1개인가", "N+1을 어떻게 찾고 고쳤나" | 결과물보다 **의사결정 과정을 설명하는 능력**이 평가됨. 특히 3번째는 이번 성능 개선을 직접 수행하면 최고의 스토리가 됨 |

### 한 줄 요약

> 제품을 상상하고 → 만들고 → 배포하고 → 실제로 쓰게 만드는 전체 사이클을 혼자 완주한 것은 분명한 상위 역량입니다. 다음 병목은 기능이 아니라 **엔지니어링 규율(테스트·데이터 계층·구조화)**이고, 마침 이 프로젝트가 그걸 연습하기 딱 좋은 크기와 실데이터를 갖고 있습니다. 새 프로젝트를 시작하지 말고, 이 프로젝트를 "잘 만든 프로젝트"에서 "잘 관리되는 프로젝트"로 승격시키는 것을 다음 목표로 삼으세요.
